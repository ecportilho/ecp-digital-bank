2. Fluxo: Pagamento com Cartão de Crédito

Sequência completa de um pagamento com cartão de crédito no FoodFlow, desde a seleção dos itens até a confirmação do pedido, incluindo split de valores e notificações entre sistemas.

sequenceDiagram
    actor M as Marina
    participant FW as Food Web
    participant FA as Food API
    participant PA as ECP Pay
    participant BA as Bank API
    participant EA as Emps API

    M->>FW: Seleciona itens + Pagar com cartão
    FW->>FA: POST /api/payments/credit-card
    FA->>PA: POST /pay/card (X-API-Key)

    alt Modo Internal
        PA->>PA: Aprovar instantaneamente
    else Modo Asaas
        PA->>PA: POST Asaas /payments (CREDIT_CARD)
    end

    PA-->>FA: transaction_id, status completed

    par Notificar Bank
        PA->>BA: POST /api/cards/purchase-by-number (Service Account JWT)
        BA->>BA: Registrar na fatura *4832
        BA-->>PA: purchaseId
    and Processar Splits
        PA->>PA: Calcular 85% restaurante + 15% plataforma
        PA->>EA: POST /webhooks/payment-received (X-Webhook-Secret)
        EA->>EA: Creditar conta PJ Brasa e Lenha
        EA-->>PA: status credited
    end

    FA->>FA: Atualizar pedido para confirmed
    FA-->>FW: status completed
    FW-->>M: Pedido confirmado!

3. Fluxo: Pagamento com Pix Copia e Cola

Fluxo assíncrono de pagamento via Pix, com geração de QR Code, aguardo de confirmação e settlement automático no modo interno ou via webhook no modo Asaas.

sequenceDiagram
    actor M as Marina
    participant FW as Food Web
    participant FA as Food API
    participant PA as ECP Pay
    participant BA as Bank API
    participant EA as Emps API

    M->>FW: Seleciona itens + Pagar com Pix
    FW->>FA: POST /api/payments/pix
    FA->>PA: POST /pay/pix (callback_url + X-API-Key)

    alt Modo Internal
        PA->>PA: Gerar QR Code mock
        PA->>PA: Agendar auto-aprovação (3s)
    else Modo Asaas
        PA->>PA: POST Asaas /payments (PIX)
        PA->>PA: GET Asaas /payments/id/pixQrCode
    end

    PA-->>FA: qr_code, qr_code_text, transaction_id
    FA-->>FW: pix_copy_paste, expires_at
    FW-->>M: Código Pix para copiar

    Note over PA: Aguardando pagamento...

    alt Modo Internal (3s)
        PA->>PA: Settlement automático
    else Modo Asaas
        PA->>PA: Webhook PAYMENT_CONFIRMED
    end

    PA->>FA: POST callback_url (payment.completed)
    FA->>FA: Atualizar pedido para confirmed

    par Debitar conta PF
        PA->>BA: POST /api/pix/debit-by-cpf (Service Account)
        BA->>BA: Debitar conta Marina
    and Processar Splits
        PA->>EA: POST /webhooks/payment-received
        EA->>EA: Creditar conta PJ
    end

    FW-->>M: Pedido confirmado!

4. Provider Pattern — Abstração de Gateway

Diagrama de classes mostrando o padrão Provider implementado no ECP Pay. A interface PaymentProvider permite trocar entre o modo interno (simulação) e o modo Asaas (gateway real) sem alterar o código consumidor.

classDiagram
    class PaymentProvider {
        <<interface>>
        +name: string
        +createPixCharge(input) PixChargeResult
        +createCardCharge(input) CardChargeResult
        +createBoleto(input) BoletoResult
        +refund(input) RefundResult
        +getTransactionStatus(id) TransactionStatus
        +parseWebhook(headers, body) WebhookEvent
    }

    class InternalAdapter {
        +name = "internal"
        -simulationDelay: number
        +createPixCharge() QR mock + auto-settlement
        +createCardCharge() Aprovação instantânea
        +createBoleto() Barcode mock
        +refund() Estorno imediato
    }

    class AsaasAdapter {
        +name = "asaas"
        -baseUrl: string
        -apiKey: string
        -ensureCustomer() customerId
        +createPixCharge() QR real + payload EMV
        +createCardCharge() Gateway real + token
        +createBoleto() FEBRABAN + PDF
        +refund() Estorno via API
    }

    class ProviderFactory {
        -instance: PaymentProvider
        -currentMode: string
        +getProvider() PaymentProvider
        +switchProvider(mode, userId) void
        +getCurrentMode() string
    }

    PaymentProvider <|.. InternalAdapter
    PaymentProvider <|.. AsaasAdapter
    ProviderFactory --> PaymentProvider
    ProviderFactory ..> InternalAdapter : creates
    ProviderFactory ..> AsaasAdapter : creates

5. Ciclo de Vida de uma Transação

Diagrama de estados mostrando todas as transições possíveis de uma transação no ECP Pay, desde a criação até os estados finais (completada, expirada, cancelada, estornada).

stateDiagram-v2
    [*] --> pending: Cobrança criada

    pending --> processing: Processando pagamento
    pending --> completed: Pagamento confirmado
    pending --> expired: Prazo expirado
    pending --> cancelled: Cancelada pelo usuário

    processing --> completed: Aprovado
    processing --> failed: Rejeitado

    completed --> refunded: Estorno total
    completed --> partially_refunded: Estorno parcial

    failed --> [*]
    expired --> [*]
    cancelled --> [*]
    refunded --> [*]
    partially_refunded --> [*]

    note right of pending: Pix e Boleto começam aqui
    note right of completed: Cartão vai direto para completed
    note left of processing: Cartão via Asaas pode passar por processing
